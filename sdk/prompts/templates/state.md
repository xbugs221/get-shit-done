# 状态模板

`.planning/STATE.md` 的模板 — 项目的持续更新记忆。

---

## 文件模板

```markdown
# 项目状态

## 项目引用

参见：.planning/PROJECT.md（更新于 [日期]）

**核心价值：**[PROJECT.md 核心价值部分的一句话摘要]
**当前聚焦：**[当前阶段名称]

## 当前位置

阶段：第 [X] 阶段，共 [Y] 阶段（[阶段名称]）
计划：当前阶段的第 [A] 个计划，共 [B] 个
状态：[准备规划 / 规划中 / 准备执行 / 进行中 / 阶段完成]
最后活动：[YYYY-MM-DD] — [发生了什么]

进度：[░░░░░░░░░░] 0%

## 性能指标

**速度：**
- 已完成计划总数：[N]
- 平均耗时：[X] 分钟
- 总执行时间：[X.X] 小时

**按阶段统计：**

| 阶段 | 计划数 | 总耗时 | 平均/计划 |
|------|--------|--------|----------|
| - | - | - | - |

**近期趋势：**
- 最近 5 个计划：[耗时列表]
- 趋势：[提升 / 稳定 / 下降]

*每次计划完成后更新*

## 积累的上下文

### 决策

决策记录在 PROJECT.md 的关键决策表中。
影响当前工作的近期决策：

- [阶段 X]：[决策摘要]
- [阶段 Y]：[决策摘要]

### 待办事项

[会话中捕获的待办想法]

暂无。

### 阻塞/关注点

[影响未来工作的问题]

暂无。

## 会话连续性

上次会话：[YYYY-MM-DD HH:MM]
停止位置：[最后完成操作的描述]
恢复文件：[.continue-here*.md 的路径（如果存在），否则为"无"]
```

<purpose>

STATE.md 是项目跨所有阶段和会话的短期记忆。

**它解决的问题：** 信息被记录在摘要、问题和决策中，但没有被系统性地使用。会话启动时缺乏上下文。

**解决方案：** 一个单一的小文件，它：
- 在每个工作流中首先被读取
- 在每次重要操作后更新
- 包含积累上下文的摘要
- 支持即时会话恢复

</purpose>

<lifecycle>

**创建：** 在 ROADMAP.md 创建后（初始化期间）
- 引用 PROJECT.md（读取它以获取当前上下文）
- 初始化空的积累上下文部分
- 将位置设为"阶段 1 准备规划"

**读取：** 每个工作流的第一步
- 进度：向用户展示状态
- 规划：指导规划决策
- 执行：了解当前位置
- 过渡：了解已完成的内容

**写入：** 每次重要操作后
- 执行：在 SUMMARY.md 创建后
  - 更新位置（阶段、计划、状态）
  - 记录新决策（详情在 PROJECT.md 中）
  - 添加阻塞/关注点
- 过渡：在阶段标记为完成后
  - 更新进度条
  - 清除已解决的阻塞
  - 刷新项目引用日期

</lifecycle>

<sections>

### 项目引用
指向 PROJECT.md 以获取完整上下文。包括：
- 核心价值（最重要的那一件事）
- 当前聚焦（哪个阶段）
- 最后更新日期（过时则触发重新读取）

Claude 直接读取 PROJECT.md 获取需求、约束和决策。

### 当前位置
我们现在在哪里：
- 第 X 阶段，共 Y 阶段 — 哪个阶段
- 第 A 个计划，共 B 个 — 阶段内的哪个计划
- 状态 — 当前状态
- 最后活动 — 最近发生了什么
- 进度条 — 整体完成度的可视化指标

进度计算：（已完成计划数）/（所有阶段的总计划数）× 100%

### 性能指标
跟踪速度以了解执行模式：
- 已完成计划总数
- 每个计划的平均耗时
- 按阶段分解统计
- 近期趋势（提升/稳定/下降）

每次计划完成后更新。

### 积累的上下文

**决策：** 引用 PROJECT.md 的关键决策表，加上近期决策摘要以便快速查看。完整的决策日志在 PROJECT.md 中。

**待办事项：** 会话中捕获的想法。
- 待办事项数量
- 如果较少则列出简要清单，较多则只显示数量

**阻塞/关注点：** 来自"下一阶段就绪情况"部分
- 影响未来工作的问题
- 前缀标注来源阶段
- 解决后清除

### 会话连续性
支持即时恢复：
- 上次会话的时间
- 上次完成的内容
- 是否有 .continue-here 文件可以恢复

</sections>

<size_constraint>

STATE.md 保持在 100 行以内。

它是一个摘要，不是归档。如果积累的上下文过大：
- 摘要中只保留 3-5 个近期决策（完整日志在 PROJECT.md 中）
- 只保留活跃的阻塞，移除已解决的

目标是"读一遍就知道我们在哪" — 如果太长，这个目标就失败了。

</size_constraint>
