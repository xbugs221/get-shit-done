# 路线图模板

`.planning/ROADMAP.md` 的模板。

## 初始路线图（v1.0 全新项目）

```markdown
# 路线图：[项目名称]

## 概述

[一段话描述从开始到完成的旅程]

## 阶段

**阶段编号：**
- 整数阶段（1、2、3）：计划中的里程碑工作
- 小数阶段（2.1、2.2）：紧急插入项（标记为 INSERTED）

小数阶段按数字顺序出现在相邻整数之间。

- [ ] **阶段 1：[名称]** - [一句话描述]
- [ ] **阶段 2：[名称]** - [一句话描述]
- [ ] **阶段 3：[名称]** - [一句话描述]
- [ ] **阶段 4：[名称]** - [一句话描述]

## 阶段详情

### 阶段 1：[名称]
**目标**：[本阶段交付什么]
**依赖**：无（第一阶段）
**需求**：[REQ-01, REQ-02, REQ-03]  <!-- 方括号可选，解析器支持两种格式 -->
**成功标准**（必须为真）：
  1. [从用户角度可观察的行为]
  2. [从用户角度可观察的行为]
  3. [从用户角度可观察的行为]
**计划**：[计划数量，例如"3 个计划"或"待定"]

计划列表：
- [ ] 01-01: [第一个计划的简要描述]
- [ ] 01-02: [第二个计划的简要描述]
- [ ] 01-03: [第三个计划的简要描述]

### 阶段 2：[名称]
**目标**：[本阶段交付什么]
**依赖**：阶段 1
**需求**：[REQ-04, REQ-05]
**成功标准**（必须为真）：
  1. [从用户角度可观察的行为]
  2. [从用户角度可观察的行为]
**计划**：[计划数量]

计划列表：
- [ ] 02-01: [简要描述]
- [ ] 02-02: [简要描述]

### 阶段 2.1：紧急修复（INSERTED）
**目标**：[在阶段之间插入的紧急工作]

**成功标准**（必须为真）：
  1. [修复达到的效果]
**计划**：1 个计划

计划列表：
- [ ] 02.1-01: [描述]

### 阶段 3：[名称]
**目标**：[本阶段交付什么]
**依赖**：阶段 2
**需求**：[REQ-06, REQ-07, REQ-08]
**成功标准**（必须为真）：
  1. [从用户角度可观察的行为]
  2. [从用户角度可观察的行为]
  3. [从用户角度可观察的行为]
**计划**：[计划数量]

计划列表：
- [ ] 03-01: [简要描述]
- [ ] 03-02: [简要描述]

### 阶段 4：[名称]
**目标**：[本阶段交付什么]
**依赖**：阶段 3
**需求**：[REQ-09, REQ-10]
**成功标准**（必须为真）：
  1. [从用户角度可观察的行为]
  2. [从用户角度可观察的行为]
**计划**：[计划数量]

计划列表：
- [ ] 04-01: [简要描述]

## 进度

**执行顺序：**
阶段按数字顺序执行：2 → 2.1 → 2.2 → 3 → 3.1 → 4

| 阶段 | 计划完成情况 | 状态 | 完成日期 |
|------|-------------|------|---------|
| 1. [名称] | 0/3 | 未开始 | - |
| 2. [名称] | 0/2 | 未开始 | - |
| 3. [名称] | 0/2 | 未开始 | - |
| 4. [名称] | 0/1 | 未开始 | - |
```

<guidelines>
**初始规划（v1.0）：**
- 阶段数量取决于粒度设置（粗粒度：3-5，标准：5-8，细粒度：8-12）
- 每个阶段交付一些有意义的成果
- 阶段可以有 1 个以上的计划（如果超过 3 个任务或涉及多个子系统则拆分）
- 计划使用命名规则：{阶段}-{计划}-PLAN.md（例如 01-02-PLAN.md）
- 不做时间估算（这不是企业级项目管理）
- 进度表由执行工作流更新
- 计划数量初始可以是"待定"，在规划期间细化

**成功标准：**
- 每个阶段 2-5 个可观察的行为（从用户角度）
- 在路线图创建期间与需求进行交叉检查
- 向下传递到计划阶段的 `must_haves`
- 在执行后由验证阶段进行验证
- 格式："用户可以[操作]"或"[事物]可用/存在"

**里程碑发布后：**
- 将已完成的里程碑折叠在 `<details>` 标签中
- 为即将到来的工作添加新的里程碑部分
- 保持连续的阶段编号（永远不要从 01 重新开始）
</guidelines>

<status_values>
- `未开始` - 尚未启动
- `进行中` - 正在进行
- `已完成` - 已完成（添加完成日期）
- `已推迟` - 推迟到以后（附原因）
</status_values>

## 按里程碑分组的路线图（v1.0 发布后）

v1.0 完成后，使用里程碑分组重新组织：

```markdown
# 路线图：[项目名称]

## 里程碑

- ✅ **v1.0 MVP** - 阶段 1-4（已发布 YYYY-MM-DD）
- 🚧 **v1.1 [名称]** - 阶段 5-6（进行中）
- 📋 **v2.0 [名称]** - 阶段 7-10（已规划）

## 阶段

<details>
<summary>✅ v1.0 MVP（阶段 1-4）- 已发布 YYYY-MM-DD</summary>

### 阶段 1：[名称]
**目标**：[本阶段交付什么]
**计划**：3 个计划

计划列表：
- [x] 01-01: [简要描述]
- [x] 01-02: [简要描述]
- [x] 01-03: [简要描述]

[... 其余 v1.0 阶段 ...]

</details>

### 🚧 v1.1 [名称]（进行中）

**里程碑目标：**[v1.1 交付什么]

#### 阶段 5：[名称]
**目标**：[本阶段交付什么]
**依赖**：阶段 4
**计划**：2 个计划

计划列表：
- [ ] 05-01: [简要描述]
- [ ] 05-02: [简要描述]

[... 其余 v1.1 阶段 ...]

### 📋 v2.0 [名称]（已规划）

**里程碑目标：**[v2.0 交付什么]

[... v2.0 阶段 ...]

## 进度

| 阶段 | 里程碑 | 计划完成情况 | 状态 | 完成日期 |
|------|--------|-------------|------|---------|
| 1. 基础 | v1.0 | 3/3 | 已完成 | YYYY-MM-DD |
| 2. 功能 | v1.0 | 2/2 | 已完成 | YYYY-MM-DD |
| 5. 安全 | v1.1 | 0/2 | 未开始 | - |
```

**说明：**
- 里程碑标识：✅ 已发布、🚧 进行中、📋 已规划
- 已完成的里程碑折叠在 `<details>` 中以提高可读性
- 当前/未来里程碑展开显示
- 连续的阶段编号（01-99）
- 进度表包含里程碑列
