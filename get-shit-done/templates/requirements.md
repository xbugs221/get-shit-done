# 需求模板

用于 `.planning/REQUIREMENTS.md` 的模板 — 定义"完成"标准的可检查需求。

<template>

```markdown

### 认证

- [ ] **AUTH-01**：用户可以使用邮箱和密码注册
- [ ] **AUTH-02**：用户注册后收到邮箱验证
- [ ] **AUTH-03**：用户可以通过邮箱链接重置密码
- [ ] **AUTH-04**：用户会话在浏览器刷新后保持

### [类别 2]

- [ ] **[CAT]-01**：[需求描述]
- [ ] **[CAT]-02**：[需求描述]
- [ ] **[CAT]-03**：[需求描述]

### [类别 3]

- [ ] **[CAT]-01**：[需求描述]
- [ ] **[CAT]-02**：[需求描述]

## v2 需求

推迟到未来发布。已跟踪但不在当前路线图中。

### [类别]

- **[CAT]-01**：[需求描述]
- **[CAT]-02**：[需求描述]

## 范围之外

明确排除。记录以防止范围蔓延。

| 功能 | 原因 |
|------|------|
| [功能] | [为什么排除] |
| [功能] | [为什么排除] |

## 可追溯性

| 需求 | 阶段 | 状态 |
|------|------|------|
| AUTH-01 | Phase 1 | 待处理 |

**覆盖率：**
```

<guidelines>

**需求格式：**
- ID：`[CATEGORY]-[NUMBER]`（AUTH-01、CONTENT-02、SOCIAL-03）
- 描述：以用户为中心、可测试、原子性
- 复选框：仅用于 v1 需求（v2 尚不可操作）

**类别：**
- 从研究 FEATURES.md 的类别中派生
- 保持与领域惯例一致
- 典型的：Authentication、Content、Social、Notifications、Moderation、Payments、Admin

**v1 与 v2：**
- v1：承诺的范围，将纳入路线图阶段
- v2：已确认但推迟，不在当前路线图中
- 将 v2 移至 v1 需要更新路线图

**范围之外：**
- 附带理由的明确排除
- 防止之后出现"为什么不包含 X？"的问题
- 研究中的反功能应附带警告放在这里

**可追溯性：**
- 初始为空，在路线图创建期间填充
- 每个需求恰好映射到一个阶段
- 未映射的需求 = 路线图缺口

**状态值：**
- Pending（待处理）：未开始
- In Progress（进行中）：阶段处于活跃状态
- Complete（完成）：需求已验证
- Blocked（阻塞）：等待外部因素

</guidelines>

<evolution>

**每个阶段完成后：**
1. 将已覆盖的需求标记为 Complete
2. 更新可追溯性状态
3. 记录任何范围发生变化的需求

**路线图更新后：**
1. 验证所有 v1 需求仍然被映射
2. 如果范围扩大则添加新需求
3. 如果范围缩小则将需求移至 v2/范围之外

**需求完成标准：**
- 需求在以下情况下为"Complete"：
  - 功能已实现
  - 功能已验证（测试通过、手动检查完成）
  - 功能已提交

</evolution>

```markdown
- [ ] **CONT-03**：用户可以编辑自己的帖子
- [ ] **CONT-04**：用户可以删除自己的帖子

- [ ] **SOCL-01**：用户可以关注其他用户
- [ ] **SOCL-02**：用户可以取消关注用户
- [ ] **SOCL-03**：用户可以点赞帖子
- [ ] **SOCL-04**：用户可以评论帖子
- [ ] **SOCL-05**：用户可以查看动态信息流（关注用户的帖子）
```
