# PROJECT.md 模板

用于 `.planning/PROJECT.md` 的模板 — 持续更新的项目上下文文档。

<template>

```markdown
# [项目名称]

## 这是什么

[当前准确的描述 — 2-3 句话。这个产品做什么，面向谁？
使用用户的语言和表述方式。当现实与此描述产生偏差时及时更新。]

## 核心价值

[最重要的那一件事。如果其他一切都失败了，这个必须能工作。
一句话，在需要权衡取舍时驱动优先级排序。]

## 需求

### 已验证

<!-- 已发布并确认有价值。 -->

（暂无 — 发布后再验证）

### 活跃

<!-- 当前范围。正在朝这些目标构建。 -->

- [ ] [需求 1]
- [ ] [需求 2]
- [ ] [需求 3]

### 范围之外

<!-- 明确的边界。包含理由以防止重新添加。 -->

- [排除项 1] — [原因]
- [排除项 2] — [原因]

## 背景

[影响实施的背景信息：
- 技术环境或生态系统
- 相关的先前工作或经验
- 用户研究或反馈主题
- 已知的待解决问题]

## 约束

- **[类型]**: [什么] — [为什么]
- **[类型]**: [什么] — [为什么]

常见类型：技术栈、时间线、预算、依赖、兼容性、性能、安全

## 关键决策

<!-- 约束未来工作的决策。在项目生命周期中持续添加。 -->

| 决策 | 理由 | 结果 |
|------|------|------|
| [选择] | [原因] | [✓ 良好 / ⚠️ 需重新审视 / — 待定] |

---
*最后更新：[日期] 在 [触发事件] 之后*
```

</template>

<guidelines>

**这是什么：**
- 产品的当前准确描述
- 2-3 句话概括它做什么以及面向谁
- 使用用户的措辞和表述
- 当产品发展超出此描述时更新

**核心价值：**
- 最重要的那一件事
- 其他一切都可以失败；这个不行
- 在需要权衡取舍时驱动优先级排序
- 很少变化；如果变了，说明是一次重大转向

**需求 — 已验证：**
- 已发布并证明有价值的需求
- 格式：`- ✓ [需求] — [版本/阶段]`
- 这些是锁定的 — 更改需要明确讨论

**需求 — 活跃：**
- 正在构建的当前范围
- 在发布和验证之前，这些都是假设
- 发布后移至已验证，如果被否定则移至范围之外

**需求 — 范围之外：**
- 关于我们不构建什么的明确边界
- 始终包含理由（防止之后重新添加）
- 包括：考虑后被拒绝的、推迟到未来的、明确排除的

**背景：**
- 影响实施决策的背景信息
- 技术环境、先前工作、用户反馈
- 已知问题或需要解决的技术债
- 出现新背景时更新

**约束：**
- 对实施选择的硬性限制
- 技术栈、时间线、预算、兼容性、依赖
- 包含"为什么" — 没有理由的约束会被质疑

**关键决策：**
- 影响未来工作的重大选择
- 在整个项目过程中做出决策时添加
- 已知结果时跟踪结果：
  - ✓ 良好 — 决策被证明是正确的
  - ⚠️ 需重新审视 — 决策可能需要重新考虑
  - — 待定 — 评估为时尚早

**最后更新：**
- 始终记录文档何时以及为何更新
- 格式：`在阶段 2 之后` 或 `在 v1.0 里程碑之后`
- 触发对内容是否仍然准确的审查

</guidelines>

<evolution>

PROJECT.md 在整个项目生命周期中不断演进。
这些规则嵌入在生成的 PROJECT.md 中（## 演进 部分），
并由 workflows/transition.md 和 workflows/complete-milestone.md 实施。

**每次阶段转换后：**
1. 需求被否定了？ → 移至范围之外并附上原因
2. 需求被验证了？ → 移至已验证并附上阶段引用
3. 出现了新需求？ → 添加到活跃
4. 需要记录决策？ → 添加到关键决策
5. "这是什么"仍然准确？ → 如有偏差则更新

**每个里程碑之后：**
1. 对所有部分进行全面审查
2. 核心价值检查 — 优先级仍然正确吗？
3. 审计范围之外 — 理由仍然有效吗？
4. 用当前状态更新背景（用户、反馈、指标）

</evolution>

<brownfield>

对于已有代码库：

1. **首先映射代码库**，通过 `/gsd:map-codebase`

2. **从现有代码推断已验证的需求**：
   - 代码库实际做了什么？
   - 建立了什么模式？
   - 什么明显在运行并被依赖？

3. **从用户收集活跃需求**：
   - 展示推断的当前状态
   - 询问他们接下来想构建什么

4. **初始化：**
   - 已验证 = 从现有代码推断
   - 活跃 = 用户对本次工作的目标
   - 范围之外 = 用户指定的边界
   - 背景 = 包含当前代码库状态

</brownfield>

<state_reference>

STATE.md 引用 PROJECT.md：

```markdown
## 项目引用

参见：.planning/PROJECT.md（更新于 [日期]）

**核心价值：** [核心价值部分的一句话]
**当前焦点：** [当前阶段名称]
```

这确保 Claude 读取当前的 PROJECT.md 上下文。

</state_reference>
</output>
